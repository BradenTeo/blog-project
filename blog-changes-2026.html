Dark Mode New Feature

What was the problem?

The site had no dark mode at all. The toggle button existed in the design but did nothing visible.

How does dark mode work in Tailwind?

Tailwind's dark mode works via a class strategy: when a dark class is added to the <html> element, any utility prefixed with dark: activates.

<!-- nuxt.config.js -->
tailwindcss: {
  config: {
    darkMode: 'class',   // ← this enables class-based dark mode
  }
}

The toggle logic (layouts/default.vue)

toggleDarkMode() {
  this.darkMode = !this.darkMode
  document.documentElement.classList.toggle('dark')  // adds/removes 'dark' on <html>
  localStorage.setItem('darkMode', this.darkMode)    // persist preference
},
mounted() {
  this.darkMode = localStorage.getItem('darkMode') === 'true'
  if (this.darkMode) document.documentElement.classList.add('dark')
}

Adding dark classes to components

Once the strategy was configured, dark mode classes were added throughout every component. For example:

<!-- Before -->
<div class="bg-white text-gray-900">

<!-- After -->
<div class="bg-white dark:bg-gray-800 text-gray-900 dark:text-gray-100">
Key insight: Dark mode classes don't replace regular classes — they stack on top. The browser applies whichever one is relevant based on the presence of the dark class on <html>.

Scrollbar Layout Shift Bug Fix

What was the problem?

When navigating between pages, the scrollbar appeared on long pages and disappeared on short ones. This caused the entire page content to shift left/right by ~15px, which is jarring to watch.

Root cause

A scrollbar takes up physical space. When it appears, it shrinks the viewport width, pushing content inward. When it disappears, content expands back out.

The fix

Force the scrollbar to always be visible — even on pages that don't need it. This reserves the space permanently.

/* assets/css/main.css */
html {
  overflow-y: scroll;
}

Short pages showing a scrollbar

This introduced a new issue: short pages showed an unnecessary scrollbar because the layout used min-h-screen plus bottom padding, which exceeded the viewport height.

Fix: switch the root to a flex column, so the main content stretches to fill remaining space without overflowing.

<!-- Root div -->
<div class="min-h-screen flex flex-col">

  <!-- Main content stretches to fill, no overflow -->
  <nuxt class="flex-1 pb-20" />

</div>

Navigation Bar Redesign Styling

Problems fixed

  • Active tab had an old-fashioned border-style: outset (the 3D "pressed button" effect from the 90s)
  • When a tab became active and bold, it was wider than its inactive state, pushing the other tabs sideways
  • The tab group was not truly centred — it was centred within the remaining space after the dark mode toggle, not within the full nav width

Fix 1 — Always-bold links (prevent width shift)

If all links are always bold, they always occupy their "maximum" width. The active state adds a background tint rather than changing font weight, so nothing shifts.

.link {
  font-weight: bold;       /* always bold — no width change on activation */
  color: rgba(255,255,255,0.65);
  padding: 3px 12px;
  border-radius: 6px;
  transition: color 0.15s ease, background-color 0.15s ease;
}

.link:hover {
  color: white;
  background-color: rgba(255,255,255,0.12);
}

.link.nuxt-link-active {
  color: white;
  background-color: rgba(255,255,255,0.2);  /* subtle white tint pill */
}

Fix 2 — True centering with absolute positioning

When a nav has items on both the left and right, justify-center on a flex-1 child centres within the leftover space — not the full nav. The correct approach is to absolutely position the centre group:

<nav class="relative h-9 flex items-center px-3">

  <!-- Centred relative to the full nav width -->
  <div class="absolute left-1/2 -translate-x-1/2 flex items-center space-x-1">
    <nuxt-link>Home</nuxt-link>
    <nuxt-link>Build Process</nuxt-link>
    <nuxt-link>Posts</nuxt-link>
  </div>

  <!-- Toggle pushed to the right -->
  <div class="ml-auto"> ... </div>

</nav>

Fix 3 — Home link always highlighted

Nuxt's nuxt-link-active class activates on any route that starts with the link's path. Since every route starts with /, the Home link was always highlighted.

Fix: add the exact prop, which switches it to use nuxt-link-exact-active (only matches / precisely).

<nuxt-link to="/" class="link" exact> Home </nuxt-link>

Footer — Fixed Position & Dialog Stacking Bug Bug Fix

Making the footer fixed

The footer was in normal document flow, meaning you had to scroll to the bottom to see it. Changed it to position: fixed so it's always visible at the bottom of the viewport.

<footer class="fixed bottom-0 left-0 right-0 z-50">

Added pb-20 to the main content so the footer doesn't cover the bottom of the page.

The stacking context bug

After fixing the footer, the feedback modal's backdrop no longer covered the nav bar — it was trapped behind it.

What is a stacking context? Certain CSS properties (like position: fixed combined with z-index) create an isolated "layer" for an element and all its children. No child can ever appear above elements that are siblings of its stacking-context ancestor, regardless of how high its own z-index is.

The BaseDialog component was rendered inside the <footer>, which had z-index: 50. Even with z-index: 200 on the dialog itself, it could never escape the footer's stacking context.

Fix: move <BaseDialogVue> outside the <footer> tag entirely, so it forms its own top-level stacking context.

<!-- Wrong: dialog trapped inside footer's stacking context -->
<footer class="fixed z-50">
  <BaseDialogVue />
</footer>

<!-- Correct: dialog is a sibling of the footer -->
<footer class="fixed z-50"> ... </footer>
<BaseDialogVue />

Scroll-to-Top Button New Feature

A new ScrollToTop.vue component was created. Key implementation details:

Scroll listener with passive flag

mounted() {
  window.addEventListener('scroll', this.handleScroll, { passive: true })
  // passive: true tells the browser this listener won't call preventDefault()
  // This allows the browser to optimise scrolling performance
},
beforeDestroy() {
  window.removeEventListener('scroll', this.handleScroll)
  // Always clean up listeners to prevent memory leaks
}

Smooth scroll to top

scrollToTop() {
  window.scrollTo({ top: 0, behavior: 'smooth' })
}

Vue transition for fade in/out

<transition name="fade">
  <button v-if="visible" @click="scrollToTop"> ↑ </button>
</transition>

<style scoped>
.fade-enter-active, .fade-leave-active { transition: opacity 0.2s; }
.fade-enter, .fade-leave-to           { opacity: 0; }
</style>

ClickableTab Component Redesign Styling

Problems with the original design

  • A separate "Show / Hide" button was used to toggle content — not intuitive
  • When the tab was open and the user scrolled, the sticky title would float over the content text with no background, making it unreadable

Making the entire header row clickable

Instead of a dedicated button, the whole header row (title + chevron) became the toggle target:

<div
  class="flex items-center justify-between cursor-pointer select-none"
  @click="tabOpened = !tabOpened"
>
  <h1><slot name="title"></slot></h1>
  <!-- rotating chevron -->
</div>

select-none prevents the browser from selecting text when the user double-clicks to toggle.

Rotating chevron

<svg
  class="transition-all duration-200"
  :class="{
    'rotate-180': tabOpened,           <!-- points up when open -->
    'group-hover:text-slate-600': !tabOpened,
  }"
>

Sticky header with solid background

The header row is made sticky only when the tab is open, and given the same background colour as the open tab — so it acts as a clean mask as content scrolls beneath it.

:class="{
  'sticky top-9 z-10 bg-orange-100 dark:bg-gray-700 pb-2 mb-2 border-b-2': tabOpened
}"
Why top-9? The nav bar is h-9 (36px). Sticky elements use top: X to define where they "stick". Setting top-9 means the tab header sticks just below the nav bar rather than sliding under it.

Modal / Feedback Dialog Styling

Backdrop blur

Added a full-screen overlay behind the dialog with a blur effect:

<div
  class="fixed inset-0 z-[150] backdrop-blur-sm bg-black/30"
  @click="tryClose"
></div>
  • inset-0 — stretches to cover the entire viewport
  • backdrop-blur-sm — blurs everything behind this element
  • bg-black/30 — 30% opacity black overlay
  • @click="tryClose" — clicking outside the dialog closes it

Why two sibling elements inside teleport?

<teleport to="body">
  <div>                 <!-- wrapper needed: teleport only accepts one root -->
    <div class="backdrop"></div>
    <dialog> ... </dialog>
  </div>
</teleport>

Link Hover Styles Styling

Plain text links throughout the site were hard to identify. A global rule was added to underline them on hover:

/* assets/css/main.css */
a:not(.link):not(.card-link):hover {
  text-decoration: underline;
  text-underline-offset: 3px;
  opacity: 0.75;
}

/* Blue tint for links in dark mode */
.dark a:not(.link):not(.card-link) {
  color: rgb(147 197 253);
}

The :not() exclusions prevent this rule from affecting the nav links (.link) and the post cards (.card-link), which have their own hover behaviour.

Feedback Form Fixes Bug Fix

Submit button not centred

<input type="submit"> is an inline element. mx-auto only centres block elements. The fix was to add display: block:

<input class="block mx-auto ..." type="submit" />

Hover bold causing layout shift

The button had hover:font-bold. Bold text is wider than normal text, so hovering caused the button to expand and shift surrounding content. Removed it entirely — color transitions are sufficient visual feedback.

Firebase URL moved to config

The Firebase URL was hardcoded in the component. Moved it to nuxt.config.js so it can be set via environment variable:

// nuxt.config.js
publicRuntimeConfig: {
  firebaseUrl: process.env.FIREBASE_URL || 'https://...'
}

// FeedbackForm.vue
fetch(this.$config.firebaseUrl, { ... })

ESLint Setup Infrastructure

ESLint (a code quality tool) was configured in .eslintrc.js but the packages were never actually installed, and there was no lint script. This was set up properly:

ESLint catches things like unused variables, missing props, and syntax errors before they become bugs in production.

CI/CD Fixes Infrastructure

What is CI/CD?

CI (Continuous Integration) automatically runs checks (like lint and build) whenever you push code to GitHub. It's a safety net that catches breakage before it reaches the live site.

Changes made to the workflow

  • Upgraded Node.js in CI from v14 → v24 to match the local development environment
  • Upgraded GitHub Actions steps from v2 → v4 (older versions were deprecated)
  • Re-enabled the lint step (it was commented out)
  • Added a nuxt generate step so CI verifies the site actually builds

How deployment works

The site is hosted on GitHub Pages. Deployment is a two-step process run locally:

npm run generate   # builds the static site into the dist/ folder
npm run deploy     # pushes dist/ to the gh-pages branch on GitHub

GitHub Pages then serves the gh-pages branch automatically.

Mobile Optimisations Styling

Footer — not fixed on mobile

A fixed footer that's always visible is useful on desktop, but on a small phone screen it eats into precious content space. The fix uses responsive Tailwind prefixes so the footer is only pinned on medium screens and above:

<!-- Only fixed from md (768px) upwards -->
<footer class="md:fixed md:bottom-0 md:left-0 md:right-0">

<!-- Main content only needs bottom padding on md+ -->
<nuxt class="flex-1 md:pb-20" />

The decorative footer gif was also hidden on mobile (hidden md:block) since it relied on absolute positioning inside the fixed footer.

Nav link wrapping — "Build Process" on two rows

On small screens the "Build Process" link was wrapping onto a second line. Two fixes combined to solve it:

  • Smaller font and padding on mobile via a CSS media query, scaling back up at sm: (640px)
  • whitespace-nowrap added directly as a Tailwind class on each <nuxt-link> — this is more reliable than putting it in scoped CSS, because Vue 2's scoped styles may not always reach child component root elements
/* Scoped CSS in default.vue */
.link {
  font-size: 0.75rem;
  padding: 3px 8px;
}

@media (min-width: 640px) {
  .link { font-size: 0.875rem; padding: 3px 12px; }
}

/* Template */
<nuxt-link class="link whitespace-nowrap">Build Process</nuxt-link>

Home page update boxes — flush with screen edge

On mobile, mx-auto on a block element with no explicit width just means "no margin" — the element fills the full viewport. Adding horizontal padding to the page's root <div> gives uniform breathing room to all content at once:

<!-- pages/index.vue root div -->
<div class="px-4 md:px-0">
  ...all content...
</div>

The overflow-x-hidden gotcha

An attempt to prevent horizontal scroll by adding overflow-x-hidden to the root layout div broke all sticky tab headers. This is a well-known CSS trap:

Why overflow breaks sticky: Setting any overflow value other than visible on an ancestor turns that element into the scroll container. Sticky elements then stick relative to that container — not the viewport. Since the root div wasn't scrolling (the page was), sticky elements appeared not to work at all.

The fix was to remove overflow-x-hidden entirely. The actual causes of horizontal overflow (nav links wrapping, etc.) were resolved directly instead.

Scroll-to-top button position

The button was positioned bottom-[76px] to clear the fixed footer on desktop. On mobile where the footer is no longer fixed, it should sit closer to the screen edge:

<button class="fixed bottom-4 md:bottom-[76px] right-4">

Feedback Form UI Redesign Styling

What changed visually

  • Removed the inner form border — the dialog already provides a container, so the extra border was visual noise
  • Labels moved above their inputs (stacked layout) instead of sitting to the side
  • Inputs have rounded corners, consistent border colour, and a focus ring on click
  • Textarea has resize-none to prevent users from dragging it to an ugly size
  • Submit changed from a bordered outline button to a filled slate button — clearer as a primary action
  • Error message now appears directly below the textarea with a red border on the field itself, rather than a floating italic line elsewhere
  • Thank-you state is warmer — two lines of text before the cat gif

Peeking image — absolute positioning trick

The original image used position: absolute; bottom: 0; left: 0 to sit at the bottom-left of the form, but it was positioned relative to the wrong ancestor. The new approach exploits how CSS resolves position: absolute:

How CSS finds the containing block: An absolutely positioned element walks up the DOM tree and positions itself relative to the first ancestor that has position set to anything other than static. If the FeedbackForm's own root div has no position, the search continues up — and finds the <dialog> element, which has position: fixed (set in BaseDialog's scoped CSS).

By not adding relative to the FeedbackForm root div, the image automatically positions itself relative to the modal dialog. Combined with bottom-0 left-0, it snaps to the modal's true bottom-left corner. The dialog's existing overflow: hidden clips the image cleanly at the modal edge, creating the peeking-from-below effect.

<img
  class="absolute bottom-0 left-0 h-36 max-h-[52%] w-auto pointer-events-none"
  src="peeking-small.png"
/>

<!-- The containing block chain:
  img (absolute) → looks for positioned ancestor
  → FeedbackForm root div (no position) → skip
  → slot wrapper div (no position)      → skip
  → dialog (position: fixed)            → FOUND ✓
-->

max-h-[52%] caps the image at just over half the dialog's height, so it peeks up from the bottom without covering the input fields. pointer-events-none ensures the image never blocks clicks on the form.

Posts Sidebar Redesign Styling

What changed

The original sidebar was a semi-transparent gray box with a fixed height and a back-arrow icon on every link. It was replaced with a clean, minimal pill-style sidebar that matches the nav bar aesthetic:

  • Removed the gray box background and fixed height
  • Added a small "POSTS" section label in uppercase tracking
  • Back-arrow chevron only on the "All Posts" link (not repeated on every item)
  • Horizontal divider between "All Posts" and the individual post links
  • Hover and active states use the same white-tint pill style as the nav bar

Light vs dark mode colors

The original sidebar used pure white-based text (rgba(255,255,255,0.65)) that worked in dark mode but was nearly invisible on the site's light orange/slate gradient background. The fix uses separate color sets for each mode via :global(.dark) in scoped CSS:

.sidebar-link {
  color: rgb(71, 85, 105);  /* slate-600 — readable on light background */
}

:global(.dark) .sidebar-link {
  color: rgba(255, 255, 255, 0.65);  /* white-tinted for dark background */
}

.sidebar-link:hover {
  color: rgb(15, 23, 42);
  background-color: rgba(0, 0, 0, 0.07);
}

:global(.dark) .sidebar-link:hover {
  color: white;
  background-color: rgba(255, 255, 255, 0.12);
}
Why :global(.dark) in scoped CSS? Vue's scoped CSS adds a unique attribute (e.g. data-v-xxxxxx) to every selector, so a rule like .dark .sidebar-link would look for a .dark element inside the component — not on the <html> element. Wrapping the ancestor in :global() opts that part out of scoping, so it correctly targets the dark class on <html>.

Cancer Journey — Color Consistency Styling

Problems fixed

  • Subtitle heading: was text-blue-800 — a blue from a completely different palette to the rest of the site's slate/gray theme. Changed to text-slate-700 dark:text-gray-200.
  • Callout blocks (dark mode): the "CVC line pros" and "IV lines cons" callouts used dark:bg-orange-950 and dark:bg-purple-950 — solid deep colours that felt visually heavy. Changed to dark:bg-orange-900/30 and dark:bg-purple-900/30, using Tailwind's opacity modifier to give a subtle tint instead.

Tailwind opacity modifier

Tailwind supports appending /N to any background color class to set its opacity, without needing a separate opacity property:

<!-- Before: solid deep background -->
<div class="dark:bg-orange-950">

<!-- After: same hue but at 30% opacity — much more muted -->
<div class="dark:bg-orange-900/30">

This keeps the orange/purple color distinction between the two callouts (which is intentional — they're contrasting two points) while making them blend more naturally into the dark mode background.
